# Guided Workflows UI/UX Redesign

Date: 2026-06-14

## Goal

Redesign the project UI around guided task workflows instead of separate feature pages. The primary problems are unclear navigation, weak loading feedback during long AI operations, and no clear landing page for completed analysis results.

The redesign covers both major user flows:

- Event insights: project selection, incident import, AI analysis, analysis result review.
- Knowledge Q&A: project selection, document upload/indexing, RAG chat with citations.

## Decisions

- Use two independent guided workflows, not one large combined flow.
- Keep old routes available or redirected for compatibility, but remove them from primary sidebar navigation.
- Use frontend staged progress for long-running analysis in this version.
- Add backend result APIs and run-level data relationships so the event result page can show the actual outputs from one analysis run.
- Treat RAG chat as the result surface for the knowledge workflow. Do not add a separate RAG result page.

## Information Architecture

Primary sidebar navigation should become:

- Event Insights Workflow
- Knowledge Q&A Workflow
- Analysis Dashboard
- Agent Runs
- System Status

Existing routes such as `/incident-upload`, `/analysis`, `/document-upload`, and `/chat` should remain available or redirect to the new workflows during migration. They should not appear as main sidebar entries.

## Event Insights Workflow

Route: `/insights/workflow`

Steps:

1. Project
   - If no project is selected, show a required project state with actions to choose or create a project.
   - If a project exists, mark this step complete.

2. Upload Incidents
   - Reuse the current CSV, Excel, and JSON ticket upload behavior.
   - Show upload results: raw rows, cleaned records, failed rows.
   - Let the user proceed to analysis after a successful upload.

3. AI Analysis
   - Run the existing incident analysis endpoint.
   - While waiting, show staged progress:
     - Classifying incidents
     - Scoring severity
     - Generating insights
     - Creating action items
   - Progress is frontend-estimated and should not show 100% until the API resolves.
   - On success, navigate to `/analysis/result/:agentRunId`.
   - On failure, keep the user on the analysis step and show retry plus a link to Agent Runs.

4. Analysis Result
   - Show the specific results from the completed agent run.
   - Include run summary, generated insights, generated action items, run metadata, and calls to action.
   - CTAs should include:
     - View full dashboard
     - View Agent Run details
     - Import more incidents

Step positioning should be automatic but not restrictive:

- No project: Step 1.
- Project with no incident data: Step 2.
- Project with unanalyzed incident records: Step 3.
- Project with recent analysis results: Step 4 available.
- Users can still go back to earlier available steps.

## Knowledge Q&A Workflow

Route: `/knowledge/workflow`

Steps:

1. Project
   - Same project-required behavior as the event workflow.

2. Knowledge Base
   - Reuse the current PDF upload behavior.
   - Show document/indexing status: document count, total pages, total chunks.
   - Show staged upload/indexing feedback:
     - Uploading document
     - Chunking document
     - Building vector index
     - Ready

3. Ask
   - Embed or refactor the existing chat UI into the workflow.
   - Chat is the result area for RAG.
   - Show knowledge base status above the chat.
   - Keep answer citations expandable.
   - Preserve Top K control.
   - While answering, show a small staged status such as retrieving knowledge base and generating answer.

If no documents are available, the workflow should stop at the Knowledge Base step and prevent direct questioning.

## Backend Design

Add run-level relationships for event analysis outputs:

- `insights.agent_run_id UUID NULL REFERENCES agent_runs(id) ON DELETE SET NULL`
- `action_items.agent_run_id UUID NULL REFERENCES agent_runs(id) ON DELETE SET NULL`

Add indexes:

- `idx_insights_agent_run_id`
- `idx_action_items_agent_run_id`

Update `run_incident_analysis` so generated insights and action items are written with the current `agent_run_id`.

Add:

`GET /agent-runs/{agent_run_id}/analysis-result`

Response should include:

- Agent run metadata: id, project_id, task_type, model_name, status, latency_ms, created_at, error_message.
- Summary from `AgentRun.output_json`: records_analyzed, needs_review, insights_created, action_items_created.
- Insights generated by this run.
- Action items generated by this run.

Add:

`GET /projects/{project_id}/workflow-status`

Response should include:

- Event status:
  - cleaned ticket count
  - analyzed ticket count
  - unanalyzed ticket count
  - latest incident analysis run id and status
- Knowledge status:
  - document count
  - total pages
  - total chunks
  - can_chat boolean

The frontend should use this status API for step positioning instead of inferring workflow state from multiple unrelated endpoints.

## Frontend Components

Create shared workflow components where possible:

- `WorkflowStepper`
  - Shows complete, current, available, and locked states.
  - Allows returning to available steps.

- `ProjectRequiredState`
  - Shared empty state when no project is selected.

- `UploadPanel`
  - Configurable upload UI for tickets and documents.
  - Supports file type labels, accepted extensions, loading text, and success summaries.

- `WorkflowStatusPanel`
  - Shows project-level workflow facts such as unanalyzed incidents, latest run, document count, pages, and chunks.

- `AnalysisProgressPanel`
  - Event-analysis-specific staged progress panel.

- `AnalysisResultPage`
  - Fetches and renders `/agent-runs/{agentRunId}/analysis-result`.

- `EventInsightsWorkflowPage`
  - Owns event workflow step state and navigation.

- `KnowledgeWorkflowPage`
  - Owns knowledge workflow step state and embeds/refactors chat behavior.

Use the existing Tailwind setup, `Card`, `Button`, and lucide icons. Keep the visual style operational and compact rather than marketing-oriented.

## Loading And Error Handling

Event analysis:

- Estimated progress should advance through stages but stop below completion until the API resolves.
- On success, show a short completed state and navigate to the result page.
- On "no records to analyze", show actions to import more incidents or open the latest result if available.
- On failure, show the failing stage, error message, retry action, and Agent Runs link.

Knowledge Q&A:

- If no project exists, stop at the project step.
- If no documents exist, stop at the knowledge base step.
- Preserve failed chat questions and show retry.
- Keep citations visible and expandable for successful answers.

Result page:

- If the run cannot be found, show a not-found state with a CTA back to the event workflow.
- If the run exists but has no linked insights or action items, show the run summary and an explanatory empty state.

## Testing Plan

Backend tests:

- Migration or schema tests confirm `agent_run_id` columns and indexes exist.
- Analysis service writes `agent_run_id` to generated insights and action items.
- Analysis result endpoint returns only data from the requested run.
- Workflow status endpoint correctly counts cleaned, analyzed, unanalyzed, document, page, and chunk totals.

Frontend tests:

- Workflow step positioning for no project, no data, ready-to-analyze, and result-available states.
- Event analysis success navigates to `/analysis/result/:agentRunId`.
- Event analysis failure stays on the analysis step and exposes retry.
- Knowledge workflow blocks chat without documents.
- Knowledge workflow enters chat when documents exist.
- Analysis result page renders summary, insights, and action items.

Manual verification:

- Create/select project -> import incidents -> run analysis -> see staged loading -> land on run result -> open dashboard.
- Create/select project -> upload PDF -> enter chat -> ask question -> inspect citations.
- Confirm sidebar only exposes the new primary workflow entries plus dashboard, Agent Runs, and system status.

## Migration Strategy

Implement in one feature branch but keep old routes available during the first version:

1. Add backend run relationships and result/status APIs.
2. Add shared workflow components.
3. Build event workflow and result page.
4. Build knowledge workflow using the existing chat behavior.
5. Update sidebar primary navigation.
6. Redirect or preserve old routes after confirming workflow coverage.

This keeps the product experience clean while reducing breakage risk from removing old pages immediately.
